# Tasks Subsystem

Core CRUD for tasks with parent/subtask hierarchy, blocking/blocked dependencies with BFS circular detection, multi-column sorting, composite filtering, auto-hide of completed tasks, and three view modes (list, board, calendar). Tasks are Zod-validated entities persisted via the repository pattern and mutated through `dataStore` actions that emit domain events for the automation engine.

## Overview

Tasks are the central entity. Every task belongs to an optional project (`projectId`) and optional section (`sectionId`). Subtasks link via `parentTaskId`. Dependencies are separate `TaskDependency` entities forming a directed graph (blocking → blocked). The subsystem exposes three view modes — table list with sections, kanban board, and calendar — plus a cross-project "All Tasks" page with nested/flat display modes.

Write path: `Component → dataStore action → taskRepository.update() → repo subscription → Zustand setState → emitDomainEvent() → automationService.handleEvent()`. Every mutation is wrapped in `beginBatch()`/`endBatch()` for aggregated toasts.

## Task Entity (Zod Schema)

| Field | Type | Notes |
|-------|------|-------|
| `id` | `string.min(1)` | UUID v4 (generated by `TaskService.create()`) |
| `projectId` | `string.min(1) \| null` | null = unlinked task |
| `parentTaskId` | `string.min(1) \| null` | null = top-level task; non-null = subtask |
| `sectionId` | `string \| null` | Section grouping within a project |
| `description` | `string.min(1).max(500)` | Required, validated by `validateTaskDescription()` |
| `notes` | `string` | Rich text (react-quill-new) |
| `assignee` | `string` | Free-text, empty = unassigned |
| `priority` | `enum('none','low','medium','high')` | Default: `none` |
| `tags` | `string[]` | Free-text tags |
| `dueDate` | `datetime \| null` | ISO string |
| `completed` | `boolean` | Completion state |
| `completedAt` | `datetime \| null` | Set on complete, cleared on uncomplete |
| `order` | `number` | Position within section (drag-drop reorder) |
| `createdAt` | `datetime` | Immutable after creation |
| `updatedAt` | `datetime` | Auto-set on every mutation |
| `lastActionAt` | `datetime \| null` | Used by TMS reinsert and "Needs Attention" sort |
| `movedToSectionAt` | `datetime \| null` | Tracks when task moved between sections (automation trigger) |
| `comments` | `any[] (optional)` | Future extensibility |
| `attachments` | `any[] (optional)` | Future extensibility |
| `customFields` | `Record<string, any> (optional)` | Future extensibility |

## Priority System

| Priority | Enum Value | Sort Weight | Badge Variant |
|----------|-----------|-------------|---------------|
| High | `'high'` | 0 (first) | `destructive` |
| Medium | `'medium'` | 1 | `default` |
| Low | `'low'` | 2 | `secondary` |
| None | `'none'` | 3 (last) | `outline` |

## Hierarchy (Parent/Subtask)

- One level of nesting: tasks have subtasks, subtasks cannot have children
- `parentTaskId` links subtask → parent; `dataStore.getSubtasks(parentId)` returns children sorted by `order`
- Cascade complete: completing a parent completes all descendants; uncompleting only uncompletes the specific task
- Cascade delete: `TaskService.cascadeDelete()` collects all descendant IDs recursively, deletes dependencies referencing any, then deletes tasks bottom-up (children first)
- Subtask expansion state is lifted to `TaskList` via `expandedTaskIds: Set<string>` for keyboard navigation awareness

Critical: Automations skip subtask events — `evaluateRules` filters out events where `parentTaskId` is non-null. By design.

## Dependencies

Directed graph: `TaskDependency { blockingTaskId, blockedTaskId }`. A task is "blocked" if any blocking task is incomplete.

| Operation | Method | Logic |
|-----------|--------|-------|
| Is blocked? | `DependencyResolverImpl.isTaskBlocked()` | Find blocking IDs → check if any incomplete |
| Circular check | `DependencyResolverImpl.hasCircularDependency()` | BFS from `blockedTaskId` through `getBlockedTasks()` — if reaches `blockingTaskId`, cycle exists |
| Add dependency | `DependencyService.addDependency()` | Validates no circular dep, then `depRepo.create()` |
| Actionable tasks | `DependencyResolverImpl.getActionableTasks()` | Filter tasks where `!isTaskBlocked()` |

Dependencies are scoped to same-project tasks in the UI (`DependencyDialog` filters by `task.projectId`).

## Sorting

`taskSortService.ts` provides pure sort functions — no store dependencies.

| Function | Purpose |
|----------|---------|
| `sortTasks(tasks, column, direction, getProjectName)` | Multi-column sort with null-safe comparisons |
| `sortByLastAction(tasks)` | Ascending by effective last action time ("Needs Attention" mode) |

### Sortable Columns

| Column ID | Comparison | Null Handling |
|-----------|-----------|---------------|
| `name` | `localeCompare` on `description` | N/A |
| `dueDate` | String compare on ISO dates | No date → sorts last |
| `priority` | Weight map (High=0, None=3) | None = weight 3 |
| `assignee` | `localeCompare` | Empty → sorts last |
| `tags` | Count first, then joined string compare | — |
| `project` | `localeCompare` on project name | "No Project" → sorts last |
| `lastAction` | `getEffectiveLastActionTime()` compare | Falls back to `createdAt` |

Column order is persisted in `appStore.columnOrder` (default: `['dueDate', 'priority', 'assignee', 'tags']`). `name` is always first, `project` only shown on All Tasks page.

## Filtering

### Filter Store (Transient — Zustand, no persist)

| Filter | Type | Default |
|--------|------|---------|
| `searchQuery` | `string` | `''` |
| `priorityFilter` | `Priority \| null` | `null` |
| `dateRangeFilter` | `{ start: Date \| null, end: Date \| null }` | Both null |
| `completionFilter` | `'all' \| 'completed' \| 'incomplete'` | `'all'` |

### Filter Pipeline (`useFilteredTasks` hook)

```
Input tasks
  → searchQuery: match description, tags, or assignee (case-insensitive)
  → priorityFilter: exact match
  → dateRangeFilter: dueDate within [start, end]
  → completionFilter: completed === true/false
  → showOnlyActionableTasks (appStore setting): DependencyResolverImpl.getActionableTasks()
  → Output filtered tasks
```

### Auto-Hide Completed Tasks

`autoHideService.ts` filters completed tasks past a configurable threshold.

| Threshold | Duration | Behavior |
|-----------|----------|----------|
| `'24h'` | 86,400,000 ms | Hide after 24 hours |
| `'48h'` | 172,800,000 ms | Hide after 48 hours |
| `'1w'` | 604,800,000 ms | Hide after 1 week |
| `'show-all'` | null | Never hide |
| `'always'` | 0 ms | Hide immediately |

Modes: In **nested** mode, subtask visibility follows parent (parent hidden → subtasks hidden). In **flat** mode, independent evaluation with active-parent exception (subtask of active parent stays visible).

## View Modes

### List View (`TaskList.tsx`)

Table-based view with collapsible sections, drag-drop task reorder, column sort/resize/reorder, and keyboard navigation integration.

| Feature | Implementation |
|---------|---------------|
| Sections | Collapsible groups, drag-drop reorder, inline rename, add/delete |
| Task rows | `TaskRow` — inline editing, subtask expansion, drag handle, priority badge |
| Column reorder | Drag-drop column headers, persisted in `appStore.columnOrder` |
| Column sort | Click header → toggle asc/desc, persisted in `appStore.sortColumn/sortDirection` |
| Keyboard nav | `useKeyboardNavigation` hook, `expandedTaskIds` lifted for `computeVisibleRows()` |
| Automation integration | `SectionContextMenuItem` for creating rules scoped to sections |
| Read-only sections | `readonlySectionIds` prop disables edit/delete/drag/collapse |

### Board View (`TaskBoard.tsx`)

Kanban board using `@dnd-kit` (core + sortable). Tasks displayed as cards within section columns. Drag-drop moves tasks between sections (`onTaskMove` callback). Sections are also sortable.

### Calendar View (`TaskCalendar.tsx`)

Monthly calendar grid using `date-fns`. Only top-level tasks shown (subtasks filtered out). Tasks placed on their `dueDate` cell. Max 3 tasks per cell with "+N more" overflow. Separate "No Due Date" section below the grid. Navigation: prev/next month, "Today" button.

## Global Tasks Page

Cross-project view via `GlobalTasksContainer` → `GlobalTasksView`.

| Display Mode | Behavior |
|-------------|----------|
| `nested` | Project tasks grouped in virtual "Tasks" section; unlinked tasks in their own sections |
| `flat` | All tasks flattened with `_flatModeParentName`, `_flatModeSubtaskCount` metadata |

Features: project column shown, "Needs Attention" sort (by `lastActionAt`), reinsert button, auto-hide threshold, `showRecentlyCompleted` toggle.

## TaskService API

| Method | Signature | Behavior |
|--------|-----------|----------|
| `create` (static) | `(data) → Task` | Factory with `uuidv4()`, timestamps, defaults |
| `completionUpdate` (static) | `(completed) → { completed, completedAt }` | Helper for completion state |
| `cascadeDelete` | `(taskId) → void` | Recursive descendant collection → delete deps → delete tasks bottom-up |
| `cascadeComplete` | `(taskId, completed, opts?) → void` | Complete: cascade to descendants. Uncomplete: single task only. Emits `task.updated` events |
| `reinsertTask` | `(taskId) → void` | Updates `lastActionAt` to now. Subtasks also bubble parent's `lastActionAt` |

`getEffectiveLastActionTime(task)`: returns `lastActionAt ?? createdAt` — used by sort and TMS.

## Domain Events Emitted

| Event | Emitted By | Trigger |
|-------|-----------|---------|
| `task.created` | `dataStore.addTask()` | After `taskRepository.create()` |
| `task.updated` | `dataStore.updateTask()` | After `taskRepository.update()`, includes `previousValues` |
| `task.deleted` | `dataStore.deleteTask()` | After `taskService.cascadeDelete()` |

Critical: `movedToSectionAt` is set in `dataStore.updateTask()` only when `sectionId` actually changes — enables automation triggers on section moves.

## Integration Points

- **Automations** — Domain events trigger rule evaluation; `SectionContextMenuItem` in TaskList for rule creation
- **Keyboard** — `useKeyboardNavigation` hook in TaskList; `expandedTaskIds` lifted for `computeVisibleRows()`
- **TMS** — `reinsertTask()` and `getEffectiveLastActionTime()` power task-ordering strategies
- **Projects** — Tasks scoped by `projectId`; `ProjectService.cascadeDelete()` deletes all project tasks
- **Sharing** — Tasks exported/imported via `ShareService`; deduplication handles ID conflicts
- **Stores** — `dataStore` is the CRUD gateway; `appStore` persists sort/column/display preferences; `filterStore` is transient
